# ToolEngine

**ONE BCG** — Multi-tenant tool-invocation platform for AI agents and human operators.

ToolEngine provides a structured, audited execution pipeline for discrete typed operations. It runs as a RESTful API or an interactive CLI, with built-in multi-tenancy, a risk-tiered human approval gate, and a six-layer MediatR pipeline.

---

## Prerequisites

- [.NET 8 SDK](https://dotnet.microsoft.com/download/dotnet/8)
- SQLite (bundled via NuGet — no separate install required for development)

---

## Getting started

### Build

```bash
git clone <repo-url>
cd onebcg-toolengine/code
dotnet build
```

### Run the API

```bash
cd src/Hosts/ToolEngine.Api
dotnet run
```

The API starts at `http://localhost:5174`. In development:
- Swagger UI: `http://localhost:5174/swagger`
- Dev token: `GET http://localhost:5174/dev/token?tenantId=acme-corp&userId=user-123`
- Health: `GET http://localhost:5174/health`

### Run the CLI

```bash
cd src/Hosts/ToolEngine.Cli
dotnet run
```

```
toolengine> invoke math calculate v1 {"a":10,"b":5}
toolengine> list
toolengine> search "add two numbers"
toolengine> exit
```

| Project | Layer | Responsibility |

| `Tools.Registry` | Tools | `IToolRegistry`, `IToolDiscovery`, namespace routing |

| `Api` | Host | ASP.NET Core Minimal API, JWT auth, approval + invocation endpoints |
| `Cli` | Host | Spectre.Console REPL, `ConsoleApprovalGate` |

## Key concepts

**Tool** — a typed, versioned handler identified by `namespace.name@version` (e.g. `hr.update-employee@v1`). Tools are plain classes; no framework coupling beyond the base class.

**Pipeline** — every invocation flows through six MediatR behaviors in order: validation → tenant auth → token budget → loop detection → approval gate → audit. Each behavior is an independent guard.

**Approval gate** — tools marked `[RequiresApproval]` are intercepted before execution. Risk tier determines routing: `Low` auto-approves; `Medium`/`High` suspend execution and notify via configurable channel (dashboard, email magic-link, OTP, or webhook); `Critical` forces OTP.

**Tenant** — all state is scoped to a tenant. Each tenant has configurable allowed namespaces, token caps, and channel preferences. `TenantId` is a string slug, not a GUID.

**Correlation** — every invocation carries a `CorrelationId` (GUID) generated by the caller. It threads through the pipeline, audit records, and approval entities for end-to-end traceability.

## Quick examples

### Invoke a tool (API)

```bash
curl -X POST http://localhost:5174/tools/math/calculate/v1/invoke \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{"a":10,"b":5}'
```

```json
{
  "correlationId": "3fa85f64-...",
  "success": true,
  "data": { "result": 15 },
  "metrics": { "durationMs": 4 }
}
```

### Invoke a tool that requires approval (API)

```bash
curl -X POST http://localhost:5174/tools/hr/update-employee/v1/invoke \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{"employeeId":"E12345","newDept":"Engineering"}'
```

```json
{
  "status": "pending_approval",
  "invocationId": "9c112b4a-...",
  "pollUrl": "/invocations/9c112b4a-.../status"
}
```

```bash
# Poll until approved or denied
curl http://localhost:5174/invocations/9c112b4a-.../status \
  -H "Authorization: Bearer {token}"
```

### Approve via magic-link token

```bash
curl -X POST "http://localhost:5174/approvals/{token}/decide?action=approve" \
  -H "Content-Type: application/json" \
  -d '{"decidedByUserId":"approver-jane"}'
```

### View pending approvals (dashboard)

```bash
curl http://localhost:5174/approvals/pending \
  -H "Authorization: Bearer {token}"
```
